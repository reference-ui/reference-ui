# Reference Types Packaging Plan

We are fighting our own architecture by trying to jump straight to a frontend
component. The next step is smaller and more structural:

ship the actual TypeScript AST bundle first.

## Current direction

The reference runtime already emits a Tasty bundle under the generated
reference output. Instead of treating that as an internal implementation detail,
we should package it as a first-class generated package:

- `@reference-ui/types`

This package will be produced by the packager, just like
`@reference-ui/react`, `@reference-ui/system`, and `@reference-ui/styled`.

## Scope

For this phase, we are **not** building a frontend component yet.

We are only doing the minimum needed to make the generated Tasty output
installable and testable as a package.

## Goal

After `ref sync`, the user project should have a generated package:

- `.reference-ui/types`
- `node_modules/@reference-ui/types -> ../../.reference-ui/types`

That package should expose the emitted Tasty bundle produced from the reference
runtime.

## Architecture decision

We should stop inventing a separate path for "reference runtime internals" and
"consumable type metadata".

The Tasty output is the product.

That means:

1. reference runtime still emits the bundle
2. packager installs that bundle as `@reference-ui/types`
3. tests verify the package contract from `reference-unit`

## Required work

### 1. Add a package definition in packager

In `packages/reference-core/src/packager`, add a new declarative package entry
for:

- `@reference-ui/types`

This should live alongside the existing package definitions in
`packages.ts`.

The new definition likely should:

- be non-bundled
- write a `package.json`
- copy files from generated output, not from CLI source
- expose the Tasty manifest entry and any emitted modules needed by the runtime

## 2. Add a packager target for types

The packager currently knows about:

- `@reference-ui/react`
- `@reference-ui/system`
- `@reference-ui/styled`

We need to extend that package set to include:

- `@reference-ui/types`

This means updating the canonical package list so install flow and symlink
creation treat `types` exactly like the other generated packages.

## 3. Define the source of truth for copied files

The source for `@reference-ui/types` should be the emitted Tasty bundle already
written by the reference runtime under generated output.

We should copy from generated output rather than rebundling it.

At minimum, the plan needs a stable package contract around:

- the manifest module
- any sibling emitted modules the manifest depends on
- emitted declaration files if they are part of the Tasty output contract

## 4. Add an entry/export contract for types

We need an explicit package surface for `@reference-ui/types`.

Open question to resolve during implementation:

- should the package export only `.` via `./manifest.js`
- or should it expose additional subpath exports for direct module access

Initial bias: keep the first version minimal and export the manifest-backed
entry point first, then widen later only if needed.

## 5. Verify it in reference-unit

Add unit coverage in `packages/reference-unit` to verify that sync produces a
usable `@reference-ui/types` package.

The first tests should stay basic and structural:

- `ref sync` creates `.reference-ui/types`
- `package.json` exists and points at the expected entry
- `node_modules/@reference-ui/types` resolves to the generated package
- the package exposes the emitted Tasty manifest/bundle

These tests should prove the package exists and is wired correctly before we
start building consumers on top of it.

## Implementation sequence

1. add `@reference-ui/types` to `packager/packages.ts`
2. define how its files are copied from generated reference Tasty output
3. update packager install expectations/tests
4. add `reference-unit` coverage for the generated package
5. only after that, start building consumers of the package

## Explicit non-goals

Not part of this step:

- frontend reference viewer components
- UI rendering for types
- higher-level browsing APIs beyond what the emitted Tasty bundle already gives us
- redesigning the reference runtime itself unless packaging exposes a real gap

## Success criteria

This phase is done when:

- `@reference-ui/types` is generated by packager
- it is installed into `.reference-ui` and symlinked into `node_modules`
- `reference-unit` has tests proving the package exists and resolves
- we can treat the Tasty bundle as a stable package artifact instead of an
  incidental internal folder
